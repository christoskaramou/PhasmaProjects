# Warbound — Design & Roadmap

Warbound is an **original** real-time strategy game in the classic 3D-RTS mold.
The reference *feel* is the late-90s/early-2000s fantasy RTS: an angled command
camera over a 3D battlefield, individually-commandable units, RPG hero units with
abilities and leveling, resource economy, and base-building. **No content from any
existing game is used** — factions, unit names, lore, art, and audio are all our own.

Player faction: **Ironhold** (disciplined human-style guard, blue/gold).
Enemy: **the Wilds** (feral raiders and beasts, red/brown).

## Engine integration (how this runs)

- Lua-only project on PhasmaEngine, run via **PhasmaPlayer**.
- PhasmaPlayer auto-runs every `.lua` in `Assets/Scripts/Player/`. `warbound.lua` is
  the sole boot script: it loads the game modules through `wb_loader` and installs a
  `hooks{ update }` driver.
- **Authored scene, script-driven dynamics (ATH pattern).** `skirmish.pescene` authors
  the full static world (terrain, scenery, gold mine, camera), every unit rig, AND the
  HUD panels as real nodes. At runtime the scripts ADOPT those nodes by name
  (`Units.adopt` / `scene.find_model`) and drive only the dynamic parts (movement,
  combat, HP, selection, abilities, and the HUD's live content) — they create no
  geometry or UI. World+units are generated by a one-shot bake (`WB_BAKE=1` →
  `Units.build` + `scene.save`); the HUD runtime_ui nodes are appended by
  `tools/build_hud.py` (they can't be script-created — the engine only renders
  runtime_ui nodes present at scene load). Render settings + key light stay in
  `World.setup_stage` (config, not hierarchy).
- **HUD adoption.** `wb_hud` finds the authored panels (`HUD_Minimap/Portrait/Command/
  Resources/Fps/Objective`), reads each one's `get_ui_rect`, and draws dynamic content
  (minimap dots, portrait + bars, command buttons, floating HP) inside it; the text
  panels (resources/FPS/objective) are updated with `node:set_ui`.
- Per-frame `dt` is derived from `engine.get_metrics().delta_ms` when the hook is
  called without one (mirrors the engine's node-script behavior).

### Engine pitfalls honored (learned from the AgainstTheHero project)

1. **Never `delete_node` mid-combat.** The renderer uses swap-and-pop storage; deleting
   a node mid-frame staling other nodes' draw constants → garbage/giant transforms.
   Dead units are **parked offstage** (huge -Y offset) and their rigs pooled for reuse.
2. **Bake scale at node creation.** Per-frame writes to a node's scale are unreliable;
   only position / rotation / material writes are safe every frame. Unit size is set once.
3. **Reuse the scene's active camera.** Adding a new camera + `set_active` does not swap
   the render camera mid-run. `wb_camera` reconfigures the existing active camera.

### No built-in picking — we roll our own

The runtime exposes no screen→world pick. `wb_camera` builds it from the camera basis
(`get_position/front/right/up`) + `get_fov`/`get_aspect`:
- **screen→ground**: build a ray through the cursor, intersect the `y=0` plane (move orders).
- **world→screen**: project a world point with the same basis math (unit picking, box
  select, floating HP bars). Using one consistent convention keeps pick and project
  self-consistent even if absolute FOV calibration drifts.

## Module map (`Assets/Scripts/game/`)

| Module | Responsibility |
|---|---|
| `wb_loader.lua`    | tiny module loader (fs.read + load), shared `WB` namespace injected into each module |
| `wb_util.lua`      | math/vec helpers, clamp/lerp, color tables, list utilities, ascii glyph fixup |
| `wb_world.lua`     | ground plane, scenery (trees/rocks/gold mine), bounds, render stage + lighting |
| `wb_camera.lua`    | angled camera rig: pan/zoom/edge-scroll; `pick_ground`, `world_to_screen` |
| `wb_units.lua`     | unit archetypes + factory (primitive rigs), spawn, park-on-death pooling, selection ring |
| `wb_selection.lua` | left-click / drag-box selection, selection rings, hover |
| `wb_orders.lua`    | issue move/attack orders, per-frame steering + separation, arrival |
| `wb_combat.lua`    | acquire targets, auto-attack, damage, death, hero XP/leveling |
| `wb_abilities.lua` | hero abilities (Warstomp AoE, cooldown + mana) |
| `wb_hud.lua`       | runtime_ui HUD: minimap, portrait + bars, command card, resource bar, floating HP |
| `wb_game.lua`      | orchestrator: owns world state, `init()` builds the match, `update(dt)` ticks subsystems |

## Roadmap

- **Slice 1 (now): Hero + Squad combat.** Camera, selection, movement, auto-combat, a
  hero with one active ability + leveling, an enemy camp to clear, full HUD. ← current
- **Slice 2: Economy + base-building.** Gather gold from the mine + lumber from trees,
  a Town Hall + Barracks, train units, food/upkeep.
- **Slice 3: Tech + abilities.** Upgrades, a second hero, more abilities, items.
- **Slice 4: Enemy AI.** A scripted opponent that builds and attacks.
- **Slice 5: Art pass.** Swap primitive rigs for CC0/original 3D models; map decoration.

Status notes and per-session handoffs live in git history and the engine's MemPalace.
